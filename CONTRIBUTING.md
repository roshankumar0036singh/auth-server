# Contributing to Auth Server

First off, thank you for considering contributing to Auth Server! It's people like you that make Auth Server such a great tool.

## Code of Conduct

This project and everyone participating in it is governed by our [Code of Conduct](./CODE_OF_CONDUCT.md). By participating, you are expected to uphold this code.

## How Can I Contribute?

### Reporting Bugs

This section guides you through submitting a bug report for Auth Server. Following these guidelines helps maintainers and the community understand your report, reproduce the behavior, and find related reports.

*   **Check the [Issues](https://github.com/roshankumar0036singh/auth-server/issues)** to see if the bug has already been reported.
*   **Use the [Bug Report Template](https://github.com/roshankumar0036singh/auth-server/issues/new?template=bug_report.md)** to provide as much information as possible.

### Suggesting Enhancements

This section guides you through submitting an enhancement suggestion for Auth Server, including completely new features and minor improvements to existing functionality.

*   **Check the [Issues](https://github.com/roshankumar0036singh/auth-server/issues)** to see if the enhancement has already been suggested.
*   **Use the [Feature Request Template](https://github.com/roshankumar0036singh/auth-server/issues/new?template=feature_request.md)** to provide a clear and concise description of the suggestion.

### Pull Requests

*   Fill in the [Pull Request Template](.github/pull_request_template.md).
*   Include screenshots and animated GIFs in your pull request whenever possible.
*   Follow the style guide (Go fmt, linting).
*   Add unit tests for any new code.

## Development Setup

### Workflow
1.  Fork the repo and create your branch from `main`.
2.  If you've added code that should be tested, add tests.
3.  If you've changed APIs, update the documentation.
4.  Ensure the test suite passes.
5.  Make sure your code lints.
6.  Issue that pull request!

## Style Guide

### Go Style Guide
*   All code should follow standard `go fmt` and `golangci-lint` guidelines.
*   Use descriptive variable names.
*   Keep functions small and focused.

### Commit Messages
*   Use the [Conventional Commits](https://www.conventionalcommits.org/) format:
    *   `feat: add new oauth provider`
    *   `fix: resolve memory leak in session manager`
    *   `docs: update contributing guide`

## Questions?
